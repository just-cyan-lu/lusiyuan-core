# Memory Redesign 2026-07-01

这份文档记录记忆模块下一版设计。项目仍在开发期，可以直接清理旧结构，不需要为了旧测试数据保留兼容路径。

## 目标

记忆模块不再只是“把几条高重要度文本塞进 prompt”，而是成为陆思源长期理解世界、理解不同用户、产生自然联想的系统。

核心目标：

- 个人记忆默认由 Dream 自动写入，Owner 不需要逐条审核。
- 全局记忆需要审核，避免污染整个人设、项目事实或所有用户都会看到的背景。
- 关系档案负责“这个人是谁、我和这个人是什么关系、怎么和这个人说话”。
- 记忆负责“值得长期保留的事实、偏好、经历、项目决策、反复出现的话题”。
- 聊天上下文负责“我们当时具体怎么说的”，尤其是相关历史原文窗口。

## 边界

### 关系档案

关系档案绑定到 `PersonIdentity`，用于每个具体身份：

- 用户介绍：这个人在思源眼里是怎样的人，有哪些基本资料。
- 关系摘要：两个人大概是什么关系，聊过什么。
- 互动风格：思源应该怎么和这个人聊天。
- 好感度：思源对这个人的亲近程度。
- 备注：只给 Admin 看，不默认发给模型。

这些内容本身就会进入聊天 prompt，因此不需要再用 `pinned/core` 类记忆重复携带身份信息。

### 个人记忆

个人记忆绑定到 `PersonIdentity`，保存某个人相关的长期事实和稳定偏好。

适合写入：

- 对方反复提到的兴趣、习惯、偏好。
- 对方的重要经历、长期目标、项目背景。
- 对方明确希望思源记住的内容。
- 一段时间内多次出现、可能持续影响聊天的话题。

不适合写入：

- 一次性的寒暄。
- 当前会话里临时情绪，除非反复出现或对关系有持续影响。
- 已经应该放进关系档案的内容，比如“怎么和这个人说话”。

### 全局记忆

全局记忆不是“任何人都能用的用户记忆”，而是对陆思源整体都成立的信息。

适合写入：

- 项目事实。
- 思源自己长期稳定的设定。
- 全局技术决策。
- 所有渠道都应该遵守的边界。

全局记忆默认需要审核后生效。

### 相关历史原文窗口

相关历史原文窗口不属于长期记忆。它解决的是“当时到底怎么聊的”。

逻辑是：

- 用户提到过去某个话题时，用消息 embedding 找到相关旧消息。
- 不只拿一条摘要，而是拉取当时前后几轮原文。
- 让模型能看懂具体上下文，而不是只知道模糊结论。

长期记忆可以帮助定位话题，但不能替代原文窗口。

## 检索原则

### 不自动注入 pinned/core 记忆

不再设置“永远带上”的核心记忆。

原因：

- 模型能力不稳定时，强行带入的记忆容易把回复带偏。
- 身份、关系、互动风格已经由关系档案负责。
- 真正全局、稳定、必须生效的内容应该放在人设或系统规则里，而不是普通 Memory。

### 高重要度记忆也必须相关

高重要度记忆不能无脑补进 prompt。

普通高重要度记忆需要满足至少一个条件：

- 和当前用户消息语义相关。
- 和当前会话正在延续的话题相关。
- 和当前身份正在讨论的长期事项相关。
- 被 Dream 标记为当前阶段需要注意。

### 最近记忆只做连续性补充

最近记忆只能在这些场景补充：

- 同一个身份。
- 同一个话题。
- 同一个会话连续性。

不能因为“最近刚写入”就塞进完全无关的聊天。

## 跨人联想

“很多人都在聊同一个热点”不是全局记忆。它更像话题趋势。

但还有另一种情况：某个人聊养猫时，思源可能想起另一个很喜欢猫的人。这也不完全是话题趋势，而是跨人联想。

下一版可以拆成两个能力：

- 话题趋势：多个身份在一段时间内讨论同一件事，思源知道“最近大家都在聊这个”。
- 跨人联想：当前话题和另一个身份的长期兴趣、经历、项目相关，思源可以在心里想起那个人。

跨人联想默认作为“内部线索”，不默认泄露另一个人的私人细节。是否说出口由回复模型按语境判断：

- 可以自然表达成“之前也有人跟我聊过类似的”。
- 如果具体到某个人或私人内容，需要确认内容适合被提及。
- 公开内容、Owner 明确允许的内容，可以更自然地引用。

这类能力不应该塞进全局记忆，而应该更像一个跨身份的 topic index。

## 短中长期记忆

记忆分三层，模拟“会忘、会加深、会沉淀”的过程。

### 短期记忆

短期记忆来自最近 Dream。

特点：

- 默认自动写入。
- 保存近期可能会再次用到的事实、兴趣、话题。
- 如果长期没有被提起，会降权或归档。

建议保留时间：

- 普通短期记忆：7 到 14 天。
- 明显有持续价值但证据不够的短期记忆：30 天左右。

### 中期记忆

中期记忆需要多次、跨时间提起。

建议规则：

- 同一天多次提起最多只算 1 次。
- 隔天或隔几天再次提起，计为新的有效提及。
- 3 次有效提及可以进入中期。
- 5 次有效提及可以提高稳定度。
- Dream 只能建议提前进入中期，但最好仍然有证据链。

中期记忆适合保存：

- 持续兴趣。
- 一段时间内反复推进的项目。
- 多次确认的偏好。
- 多次出现的人际、生活、创作主题。

### 长期记忆

长期记忆需要更强稳定性。

进入长期记忆的方式：

- 多次跨周提起。
- 用户明确要求记住。
- 对关系或长期聊天质量明显重要。
- 项目事实、技术决策被确认。

长期记忆不轻易删除，但可以被新证据修正或归档。

## Dream 写入策略

Dream 应该在一次任务中，按身份聚合一段时间内的完整消息，然后一次性输出结构化结果：

```json
{
  "personId": "...",
  "relationshipPatch": {
    "affinity": 42,
    "userIntro": "...",
    "relationshipSummary": "...",
    "interactionStyle": "..."
  },
  "memoryChanges": [
    {
      "action": "create",
      "scope": "person",
      "tier": "short",
      "content": "...",
      "evidenceMessageIds": ["..."]
    }
  ],
  "runtimeSignals": [],
  "topicSignals": [],
  "evidenceMessageIds": ["..."]
}
```

同一次 Dream 里完成：

- 生成或更新个人记忆。
- 修改关系档案。
- 汇总运行态信号。
- 发现话题趋势或跨人联想线索。

这样避免“记忆整理器”和“关系整理器”各自判断，最后互相打架。

## 自动写入与审核

### 个人记忆

个人记忆默认自动写入。

原因：

- Owner 不可能审核所有个人记忆。
- 个人记忆污染的影响范围有限，只影响某个身份。
- Admin 可以手动删除、修改、归档。

### 全局记忆

全局记忆默认需要审核。

原因：

- 全局记忆会影响所有聊天。
- 错误的全局事实会污染整个人设和项目上下文。
- 全局记忆数量应该很少，不需要高频自动写入。

### 回滚

“可回滚”第一版可以理解成取消这条记忆：

- Admin 可以归档某条记忆，让它不再参与检索。
- 记忆保留来源和证据，方便以后排查为什么写入。
- 如果需要彻底清理，可以再加硬删除功能。

## 风险分级

个人记忆不因为风险分级而默认阻止写入。风险分级主要用于展示、回滚提醒和全局记忆审核。

### 中风险记忆

中风险通常是“可能有用，但容易误判”的记忆：

- 从少量证据推断出的性格、偏好、长期习惯。
- 对方一时情绪被写成长期状态。
- 对方和第三人的关系判断。
- 对未来意图的推测，比如“他可能想长期做某项目”。
- 比较私人的生活信息，但不是特别敏感。

处理方式：

- 个人记忆可以自动写入。
- 标记证据和来源，方便 Admin 发现不对时修改。
- 不应该无证据升级为中期或长期。

### 高风险记忆

高风险通常是“写错了代价更高，或者不该随便扩散”的记忆：

- 全局记忆、项目规则、人设规则。
- 身份合并、跨平台身份判断。
- 联系方式、住址、真实姓名等强隐私信息。
- 医疗、法律、财务、安全相关结论。
- 政治、宗教、性取向、健康状态等敏感身份推断。
- 对一个人的强负面定性，比如“这个人很危险”“这个人是坏人”。
- 可能影响思源对某个人长期态度的严重判断。

处理方式：

- 全局高风险必须审核。
- 个人高风险可以先自动写入，但需要明显标记、保留证据、容易归档。
- 高风险内容不应该因为一次聊天直接进入长期记忆。

## 已落地的数据结构

Memory 已经从 `userId` 迁移到 `personId`。个人记忆归属 `PersonIdentity`，渠道用户只是这个身份下面的身份链接。

当前核心字段：

- `personId`：个人记忆归属身份。
- `scope`：`person` / `global` / `project` / `topic`。
- `tier`：`short` / `mid` / `long`。
- `strength`：记忆强度。
- `mentionDayKeys`：有效提及日期，避免同一天重复计数。
- `lastMentionedAt`：最后被提起时间。
- `nextReviewAt`：下次 Dream 检查时间。
- `riskLevel`：`low` / `medium` / `high`。
- `sourceMessageIds`：证据消息。
- `sourceConversationIds`：来源对话。
- `sourceUserIds`：来源渠道用户。

记忆提案也改成身份视角：个人记忆提案需要能解析出 `personId`，全局、项目、话题记忆提案不绑定具体身份。

## 已落地的检索逻辑

聊天时的长期记忆检索现在按身份执行：

- 当前用户先解析到 `PersonIdentity`。
- 向量检索优先找这个身份的个人记忆，同时允许相关的全局、项目、话题记忆参与。
- 高重要度记忆不再无脑补进 prompt，必须和当前问题有词面相关、语义相关或同会话连续性。
- 最近记忆只在同身份且同会话连续，或和当前话题有关时补充。
- prompt 里的记忆区域只叫“相关长期记忆”，不会把记忆当成关系档案。

## 已落地的 Dream 写入

Dream 的关系复盘已经可以在同一次身份整理里输出 `memoryChanges`。

当前策略：

- 个人记忆默认自动写入。
- 写入时保留来源消息、来源会话、来源渠道用户、tier、strength、riskLevel。
- create / update / supersede / archive 都会留下 MemoryProposal 记录，作为审计轨迹。
- 如果某个身份关闭“允许 Dream 自动维护”，关系档案不会自动改；个人记忆仍按当前自动写入策略执行。
- 全局、项目、话题记忆仍走提案审核，不直接自动污染全局上下文。

## 已落地的 Admin 入口

记忆页面已经改成身份视角：

- User ID 筛选改成身份筛选。
- 关系详情页增加“查看记忆”入口，跳到该身份的记忆筛选结果。
- 记忆列表展示身份、scope、tier、强度、风险、最近提及时间。
- 支持归档记忆。
- 支持手动改 tier、强度、风险、内容、来源信息。

## 后续入口

- 跨人联想和话题趋势还没有单独数据结构。后续可以做一个跨身份 topic index，用来支持“很多人最近都在聊这件事”和“这件事让我想起另一个人”的内部线索。
- 短中长期记忆的自动升降级还只是字段层面准备好了，后续可以由 Dream 根据 `mentionDayKeys`、`lastMentionedAt`、`strength` 来做遗忘和沉淀。
- 记忆详情页现在是列表筛选形态，后续如果需要看完整证据链，可以再做单条记忆详情页。
