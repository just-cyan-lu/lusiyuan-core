# Memory Redesign 2026-07-01

这份文档记录记忆模块下一版设计。项目仍在开发期，可以直接清理旧结构，不需要为了旧测试数据保留兼容路径。

## 目标

记忆模块不再只是“把几条高重要度文本塞进 prompt”，而是成为陆思源长期理解世界、理解不同用户、产生自然联想的系统。

核心目标：

- 个人记忆默认由 Dream 自动写入，Owner 不需要逐条审核。
- 全局记忆需要审核，避免污染整个人设、项目事实或所有用户都会看到的背景。
- 关系档案负责“这个人是谁、我和这个人是什么关系、怎么和这个人说话”。
- 记忆负责“值得长期保留的事实、偏好、经历、项目决策、反复出现的话题”。
- 聊天上下文负责“我们当时具体怎么说的”，尤其是相关历史原文窗口。

## 边界

### 关系档案

关系档案绑定到 `PersonIdentity`，用于每个具体身份：

- 用户介绍：这个人在思源眼里是怎样的人，有哪些基本资料。
- 关系摘要：两个人大概是什么关系，聊过什么。
- 互动风格：思源应该怎么和这个人聊天。
- 好感度：思源对这个人的亲近程度。
- 备注：只给 Admin 看，不默认发给模型。

这些内容本身就会进入聊天 prompt，因此不需要再用 `pinned/core` 类记忆重复携带身份信息。

### 个人记忆

个人记忆绑定到 `PersonIdentity`，保存某个人相关的长期事实和稳定偏好。

适合写入：

- 对方反复提到的兴趣、习惯、偏好。
- 对方的重要经历、长期目标、项目背景。
- 对方明确希望思源记住的内容。
- 一段时间内多次出现、可能持续影响聊天的话题。

不适合写入：

- 一次性的寒暄。
- 当前会话里临时情绪，除非反复出现或对关系有持续影响。
- 已经应该放进关系档案的内容，比如“怎么和这个人说话”。

### 全局记忆

全局记忆不是“任何人都能用的用户记忆”，而是对陆思源整体都成立的信息。

适合写入：

- 项目事实。
- 思源自己长期稳定的设定。
- 全局技术决策。
- 所有渠道都应该遵守的边界。

全局记忆默认需要审核后生效。

### 相关历史原文窗口

相关历史原文窗口不属于长期记忆。它解决的是“当时到底怎么聊的”。

逻辑是：

- 用户提到过去某个话题时，用消息 embedding 找到相关旧消息。
- 不只拿一条摘要，而是拉取当时前后几轮原文。
- 让模型能看懂具体上下文，而不是只知道模糊结论。

长期记忆可以帮助定位话题，但不能替代原文窗口。

## 检索原则

### 不自动注入 pinned/core 记忆

不再设置“永远带上”的核心记忆。

原因：

- 模型能力不稳定时，强行带入的记忆容易把回复带偏。
- 身份、关系、互动风格已经由关系档案负责。
- 真正全局、稳定、必须生效的内容应该放在人设或系统规则里，而不是普通 Memory。

### 高重要度记忆也必须相关

高重要度记忆不能无脑补进 prompt。

普通高重要度记忆需要满足至少一个条件：

- 和当前用户消息语义相关。
- 和当前会话正在延续的话题相关。
- 和当前身份正在讨论的长期事项相关。
- 被 Dream 标记为当前阶段需要注意。

### 最近记忆只做连续性补充

最近记忆只能在这些场景补充：

- 同一个身份。
- 同一个话题。
- 同一个会话连续性。

不能因为“最近刚写入”就塞进完全无关的聊天。

## 跨人联想

“很多人都在聊同一个热点”不是全局记忆。它更像话题趋势。

但还有另一种情况：某个人聊养猫时，思源可能想起另一个很喜欢猫的人。这也不完全是话题趋势，而是跨人联想。

下一版可以拆成两个能力：

- 话题趋势：多个身份在一段时间内讨论同一件事，思源知道“最近大家都在聊这个”。
- 跨人联想：当前话题和另一个身份的长期兴趣、经历、项目相关，思源可以在心里想起那个人。

跨人联想默认作为“内部线索”，不默认泄露另一个人的私人细节。是否说出口由回复模型按语境判断：

- 可以自然表达成“之前也有人跟我聊过类似的”。
- 如果具体到某个人或私人内容，需要确认内容适合被提及。
- 公开内容、Owner 明确允许的内容，可以更自然地引用。

这类能力不应该塞进全局记忆，而应该更像一个跨身份的 topic index。

## 临时、短期、中期、长期记忆

记忆分四层，模拟“先暂存、再确认、再加深、再沉淀”的过程。

这里的过期窗口不按自然日计算，而按“活跃聊天日”计算。

如果某个人和思源十天没聊天，这十天不会推进遗忘计数。只有这个身份当天真的有 user 消息，才算一个活跃聊天日。

同一天同一条记忆最多只算 1 次有效提及。是否真的再次触发，必须由 Dream/整理器根据上下文判断，不能只靠关键词碰巧出现。

### 临时记忆

临时记忆来自最近 Dream。

特点：

- 默认自动写入。
- 保存刚从一段聊天里整理出来、但还没确认持续有用的信息。
- 检索时权重最低，只在语义相关时作为弱参考。
- 如果窗口期内没有再次被有效提起，会归档。

已落地规则：

- 临时窗口为 5 个活跃聊天日。
- 新个人记忆默认从临时开始。
- 首次写入不算“再次触发”，`tierMentionCount` 从 0 开始。
- 临时记忆只要后续被有效提及 1 次，就升为短期。
- 如果 5 个活跃聊天日内没有再次有效提及，会自动归档。

### 短期记忆

短期记忆是已经被再次触发过的记忆。

已落地规则：

- 短期窗口为 10 个活跃聊天日。
- 短期记忆进入短期后重新计数。
- 短期记忆在窗口内累计 5 次有效提及，会升为中期。
- 如果 10 个活跃聊天日内没有再次有效提及，会降回临时。

这里没有“弱短期/普通短期”的隐藏分类。

### 中期记忆

中期记忆需要多次、跨时间提起。

建议规则：

- 同一天多次提起最多只算 1 次。
- 隔天或隔几天再次提起，计为新的有效提及。
- 中期记忆进入中期后重新计数。
- Dream 不输出 tier，系统会按有效提及次数重新计算。

已落地规则：

- 中期窗口为 60 个活跃聊天日。
- 中期记忆进入中期后重新计数。
- 中期记忆在窗口内累计 5 次有效提及，会升为长期。
- 如果 60 个活跃聊天日内没有再次有效提及，会降回短期。

中期记忆适合保存：

- 持续兴趣。
- 一段时间内反复推进的项目。
- 多次确认的偏好。
- 多次出现的人际、生活、创作主题。

### 长期记忆

长期记忆需要更强稳定性。

进入长期记忆的方式：

- 中期记忆在窗口内再次累计 5 次有效提及。
- 用户明确要求记住。
- 对关系或长期聊天质量明显重要。
- 项目事实、技术决策被确认。

长期记忆不轻易删除，但可以被新证据修正或归档。

已落地规则：

- 长期窗口为 365 个活跃聊天日。
- 长期记忆仍然需要被有效提及。
- 如果 365 个活跃聊天日内没有再次有效提及，会降回中期。
- LLM 不输出 tier；证据不足时，系统不会直接把记忆升为长期。

## Dream 写入策略

Dream 应该在一次任务中，按身份聚合一段时间内的完整消息，然后一次性输出结构化结果：

```json
{
  "personId": "...",
  "relationshipPatch": {
    "affinity": 42,
    "userIntro": "...",
    "relationshipSummary": "...",
    "interactionStyle": "..."
  },
  "memoryChanges": [
    {
      "action": "create",
      "scope": "person",
      "content": "...",
      "evidenceMessageIds": ["..."]
    }
  ],
  "runtimeSignals": [],
  "topicSignals": [],
  "evidenceMessageIds": ["..."]
}
```

同一次 Dream 里完成：

- 生成或更新个人记忆。
- 修改关系档案。
- 汇总运行态信号。
- 发现话题趋势或跨人联想线索。

这样避免“记忆整理器”和“关系整理器”各自判断，最后互相打架。

## 自动写入与审核

### 个人记忆

个人记忆默认自动写入。

原因：

- Owner 不可能审核所有个人记忆。
- 个人记忆污染的影响范围有限，只影响某个身份。
- Admin 可以手动删除、修改、归档。

### 全局记忆

全局记忆默认需要审核。

原因：

- 全局记忆会影响所有聊天。
- 错误的全局事实会污染整个人设和项目上下文。
- 全局记忆数量应该很少，不需要高频自动写入。

### 回滚

“可回滚”第一版可以理解成取消这条记忆：

- Admin 可以归档某条记忆，让它不再参与检索。
- 记忆保留来源和证据，方便以后排查为什么写入。
- 如果需要彻底清理，可以再加硬删除功能。

## 风险分级

个人记忆不因为风险分级而默认阻止写入。风险分级主要用于展示、回滚提醒和全局记忆审核。

### 中风险记忆

中风险通常是“可能有用，但容易误判”的记忆：

- 从少量证据推断出的性格、偏好、长期习惯。
- 对方一时情绪被写成长期状态。
- 对方和第三人的关系判断。
- 对未来意图的推测，比如“他可能想长期做某项目”。
- 比较私人的生活信息，但不是特别敏感。

处理方式：

- 个人记忆可以自动写入。
- 标记证据和来源，方便 Admin 发现不对时修改。
- 不应该无证据升级为中期或长期。

## 已落地的数据结构

Memory 已经从 `userId` 迁移到 `personId`。个人记忆归属 `PersonIdentity`，渠道用户只是这个身份下面的身份链接。

当前核心字段：

- `personId`：个人记忆归属身份。
- `scope`：`person` / `global` / `project` / `topic`。
- `tier`：`temp` / `short` / `mid` / `long`。
- `tierMentionCount`：当前层级内有效提及次数。
- `tierEnteredAt`：进入当前层级的时间。
- `mentionDayKeys`：已经计数过的有效提及日期，避免同一天重复计数。
- `lastMentionedAt`：最后被提起时间。
- `sourceMessageIds`：证据消息。

记忆库只保存会参与检索、展示和导出的核心信息，不再为审核、风险打分或重要度排序保留额外字段。

## 已落地的检索逻辑

聊天时的记忆检索现在按身份执行：

- 当前用户先解析到 `PersonIdentity`。
- 向量检索优先找这个身份的个人记忆，同时允许相关的全局、项目、话题记忆参与。
- 不再按重要度或最近时间无脑补记忆。
- 不再做本地关键词 rerank，也不按同会话补充旧记忆。
- prompt 里的记忆区域只叫“相关记忆”，不会把记忆当成关系档案。
- 检索排序会优先长期/中期记忆，短期次之，临时记忆权重最低；但所有层级都必须和当前话题相关才会进入 prompt。
- 每条记忆进 prompt 时会带更新时间；如果内容冲突，以更新时间更新、证据更明确的记忆为准。
- 设置页只保留 `MEMORY_RETRIEVAL_ENABLED` 和 `MEMORY_FINAL_TOP_K`。候选召回数量、语义阈值和最终上下文裁剪由系统内部固定策略处理。

## 已落地的 Dream 写入

Dream 的关系复盘已经可以在同一次身份整理里输出 `memoryChanges`。

当前策略：

- 个人记忆默认自动写入。
- 写入时保留来源消息、tier、tierMentionCount、mentionDayKeys 和 lastMentionedAt。
- create / update / supersede / archive 直接修改 Memory。
- 如果某个身份关闭“允许 Dream 自动维护”，关系档案不会自动改；个人记忆仍按当前自动写入策略执行。
- 全局、项目、话题记忆由 Deep Sleep 直接写入；后续如果发现污染，直接在记忆库归档或修改。
- Dream 每次处理某个身份前，会复查这个身份到期的个人记忆。到期后直接降级或归档，不再额外写审核记录。

## 已落地的 Admin 入口

记忆页面已经改成身份视角：

- User ID 筛选改成身份筛选。
- 关系详情页增加“查看记忆”入口，跳到该身份的记忆筛选结果。
- 记忆列表展示身份、scope、tier、本层提及次数、最近提及时间。
- 支持归档记忆。
- 支持手动改 tier、内容和来源消息。

## 后续入口

- 跨人联想和话题趋势还没有单独数据结构。后续可以做一个跨身份 topic index，用来支持“很多人最近都在聊这件事”和“这件事让我想起另一个人”的内部线索。
- 记忆详情页现在是列表筛选形态，后续如果需要看完整证据链，可以再做单条记忆详情页。
