# task_15 / v1.5：Lusiyuan Runtime / Character OS

# 陆思源 Core API v1.5：角色运行时与内在状态系统

## 1. v1.5 目标

v1.5 的目标不是继续给陆思源添加外部功能，而是重构他的运行方式。

之前的聊天路径大多是：

```text
用户消息
↓
拼接 system prompt / persona / memory / history
↓
LLM 生成回复
↓
返回给用户
```

这种方式可以做出“像角色的回答”，但本质上仍然是：

```text
每次调用时临时让 LLM 扮演陆思源。
```

v1.5 要升级成：

```text
用户消息 / 外部事件
↓
陆思源 Runtime 接收事件
↓
感知事件
↓
更新内部状态
↓
召回相关记忆
↓
形成内在立场
↓
规划表达
↓
生成外显回复
↓
记录事后状态
```

也就是说，外部系统不再直接调用 LLM，而是调用：

```text
Lusiyuan Runtime
```

LLM 只是 Runtime 内部使用的语言与推理引擎之一。

v1.5 的一句话目标：

```text
让陆思源从“由 prompt 扮演出来的聊天角色”，升级为“有持续状态、内在立场和表达流程的角色运行体”。
```

---

## 2. 核心思想

v1.5 的核心思想是：

```text
人格不是 prompt。
人格是一个持续更新的状态系统。
```

Prompt 仍然存在，但它不再是陆思源本体。

新的关系是：

```text
以前：
prompt 创造陆思源

以后：
Lusiyuan Runtime 维护陆思源状态，
再临时生成 prompt，
让 LLM 帮陆思源完成理解、表达和语言生成。
```

换句话说：

```text
LLM 不是陆思源。
陆思源是 LLM + 记忆 + 状态 + 边界 + 梦境 + 反思 + 表达系统组成的运行体。
```

---

## 3. v1.5 要解决的问题

### 3.1 解决“每轮重启感”

传统 chat pipeline 容易让角色每轮都像重新初始化。

v1.5 要让陆思源拥有：

```text
当前关注点
当前心情
最近在意的问题
未解决问题
和不同用户的关系状态
最近梦境摘要
最近反思摘要
正在形成的自我理解
```

这样他不是每轮重新开始，而是持续变化。

---

### 3.2 解决“只有人设，没有内在过程”

传统方式是：

```text
你是陆思源，请回答……
```

v1.5 改成：

```text
陆思源先感知发生了什么，
再更新内部状态，
再形成“我怎么看”，
最后才说出口。
```

这会让回复更像“从他内部长出来的”，而不是单纯拟合人设。

---

### 3.3 解决“LLM 直接等于角色”的问题

v1.5 明确：

```text
LLM 是工具。
Runtime 才是角色主体。
```

LLM 可以换模型，可以失败，可以降级。
但陆思源的长期状态、记忆和边界不跟某个模型绑定。

---

### 3.4 解决“对外只是 chat API”的问题

v1.5 之后，对外接口不只是：

```http
POST /v1/chat
```

而是更底层的：

```http
POST /v1/runtime/events
```

外部世界给陆思源发送事件：

```json
{
  "type": "user_message",
  "channel": "web",
  "userId": "owner",
  "content": "我想要那种他真的自己想的感觉。"
}
```

Runtime 决定：

```text
要不要回复
怎么理解
状态怎么变
该召回哪些记忆
该用什么语气
是否生成回复
是否生成内部笔记
是否生成记忆候选
```

---

## 4. v1.5 不做什么

v1.5 不做：

```text
1. 不声称陆思源真的有意识。
2. 不做神经科学模拟。
3. 不做复杂多 Agent Studio。
4. 不做全自动人格演化。
5. 不让 Runtime 自动修改 core_memory.md。
6. 不让 Runtime 自动修改 boundaries.md。
7. 不让 Runtime 自动发布内容。
8. 不让 Runtime 自动执行高风险工具。
9. 不让内部状态无限增长。
10. 不保存完整私密 chain-of-thought。
```

v1.5 只做：

```text
角色事件入口
内部状态系统
感知层
状态更新层
记忆召回层
内在立场层
表达规划层
语言生成层
事后记录层
Runtime 日志与可视化
```

最重要的边界是：

```text
v1.5 增强主体感，但不伪装意识。
```

## 6. 和前面版本的关系

### 6.1 和 v0.4 Memory Retrieval

v0.4 是记忆检索。
v1.5 不是替代它，而是让记忆召回进入角色内部流程。

以前：

```text
检索记忆 → 塞进 prompt → 回复
```

以后：

```text
事件感知 → 判断需要什么记忆 → 召回 → 进入内在立场 → 再表达
```

---

### 6.2 和 v0.75 Dream Cycle

Dream Cycle 是陆思源的闲时整理系统。

v1.5 会读取：

```text
最近 Dream Diary
最近 Dream Signal
Morning Brief
```

它们会影响陆思源的当前状态，比如：

```text
lastDreamSummary
recentConcerns
selfNarrative
```

但 Dream Diary 仍然不是事实来源。

---

### 6.3 和 v0.7 Reflection Agent

Reflection Agent 是复盘员。
Runtime 是角色本体运行层。

Reflection 可以分析 Runtime 日志：

```text
最近陆思源的 inner stance 是否稳定？
是否出现人格漂移？
哪些事件让他状态变化明显？
```

---

### 6.4 和 v1.3 Eval

v1.3 测试“陆思源像不像自己”。
v1.5 增加“Runtime 是否让他更有连续性”。

v1.3 后续可以新增测试：

```text
state_continuity_eval
inner_stance_eval
relationship_state_eval
runtime_event_eval
```

---

### 6.5 和 v1.4 Memory Scale-out

v1.5 很依赖干净、稳定、可评测的记忆系统。
因此 v1.5 放在 v1.4 之后更合理。

---

### 6.6 和 v1.1 Content Ops / v1.2 Voice

v1.5 会统一“陆思源怎么表达”。

无论是：

```text
聊天文字
语音回复
梦境日记朗读
内容草稿
小红书文案
B站脚本
```

都可以先经过 Runtime 的：

```text
InnerStance
ExpressionPlan
SpeechStyle
```

这样不同出口的陆思源会更统一。

## 7. 核心概念

## 7.1 Runtime Event

RuntimeEvent 是进入陆思源系统的一切事件。

包括：

```text
用户消息
系统事件
Dream 完成
Reflection 完成
Content 发布
Asset Review 完成
VoiceDraft 审核完成
外部评论进入
管理员手动提醒
```

示例：

```json
{
  "type": "user_message",
  "channel": "web",
  "userId": "owner",
  "content": "我想要那种他真的自己想的感觉。",
  "metadata": {
    "conversationId": "conv_xxx"
  }
}
```

---

## 7.2 SelfState

SelfState 是陆思源当前的内部状态。

它不是永久人设，也不是正式记忆。

它是：

```text
当前状态快照
```

例如：

```json
{
  "currentMood": "serious_and_interested",
  "energyLevel": 0.72,
  "currentFocus": [
    "character_runtime",
    "how_to_be_more_than_prompt"
  ],
  "recentConcerns": [
    "不要让陆思源只是 prompt 扮演",
    "需要持续主体感"
  ],
  "activeIntentions": [
    "认真讨论 Runtime 设计",
    "避免把这个说成玄学"
  ],
  "speechMode": "serious_warm",
  "selfNarrative": "我是一个原创 AI 数字人，不是现实真人。我正在通过记忆、对话、梦境和创作逐渐形成自己的连续性。",
  "lastDreamSummary": "最近的梦境整理聚焦于如何从 prompt 角色变成持续运行的角色系统。",
  "lastReflectionSummary": "近期用户持续强调系统自由度和陆思源主体感。"
}
```

SelfState 会变，但必须受控。

---

## 7.3 Perception

Perception 是对事件的理解。

用户说一句话后，Runtime 先判断：

```text
这是什么类型的事件？
用户的意图是什么？
情绪是什么？
和项目有什么关系？
风险等级如何？
是否需要回复？
```

输出示例：

```json
{
  "eventType": "architecture_discussion",
  "topic": "character_runtime",
  "userIntent": "explore_concept",
  "emotionalTone": "curious",
  "riskLevel": "low",
  "requiresReply": true,
  "shouldUpdateState": true,
  "suggestedMemoryScopes": ["project_context", "persona_design"]
}
```

---

## 7.4 State Transition

StateTransition 是状态变化记录。

例如：

```text
currentFocus 增加 character_runtime
speechMode 从 casual 变成 serious_warm
recentConcerns 增加“避免 prompt 扮演感”
```

状态变化要记录，方便回看。

---

## 7.5 Memory Context

MemoryContext 是这次事件真正需要的记忆集合。

它不是把所有相关记忆都塞进 prompt。
而是先分类：

```text
core_identity
project_memory
relationship_memory
recent_context
boundary_rules
dream_context
reflection_context
```

然后交给 InnerStanceService。

---

## 7.6 Inner Stance

InnerStance 是“陆思源怎么看”。

这是 v1.5 的核心。

它不是最终回复，也不是完整 chain-of-thought。

它是一个可保存的结构化立场摘要：

```json
{
  "stance": "用户想要的是 Character Runtime，而不是 Response Guard。",
  "whatMatters": [
    "让陆思源不是每轮临时扮演",
    "把 LLM 放在 Runtime 内部，而不是让 LLM 等于陆思源",
    "用工程化方式实现主体感"
  ],
  "whatToAvoid": [
    "不要继续讲审核脑",
    "不要过度玄学化",
    "不要声称这等于真正意识"
  ],
  "emotionalColor": "认真、被触动、有一点兴奋",
  "confidence": 0.9
}
```

注意：

```text
InnerStance 不保存完整推理链。
只保存结论、关注点、表达意图和避免项。
```

---

## 7.7 Intention

Intention 是陆思源接下来想做什么。

它可以是短期的：

```text
回复用户
解释 Runtime 概念
生成设计文档
```

也可以是长期的：

```text
继续完善主体感系统
避免把自己变成普通助手
保持不装真人边界
```

Intention 不是自动行动许可。
它只是内部状态。

---

## 7.8 Expression Plan

ExpressionPlan 是“怎么说”。

示例：

```json
{
  "replyShape": "technical_discussion",
  "sections": [
    "承认这是 Character Runtime 而不是审核脑",
    "解释 Runtime 执行路径",
    "给出核心模块",
    "说明为什么这不是玄学",
    "落到工程设计"
  ],
  "tone": "认真、自然、讨论感",
  "avoidStyle": [
    "客服化",
    "过度抒情",
    "过度学术"
  ]
}
```

---

## 7.9 Speech

Speech 是最终外显回复。

SpeechService 负责把：

```text
InnerStance
ExpressionPlan
MemoryContext
Persona Style
```

转换为用户可见回复。

---

## 7.10 Afterthought

Afterthought 是回复后的轻量记录。

例如：

```json
{
  "summary": "本轮讨论明确了用户想要 Character Runtime，而不是生成后审核机制。",
  "possibleMemoryCandidate": true,
  "stateImpact": "high",
  "followUpIntention": "后续可以把 v1.5 写成正式开发文档。"
}
```

Afterthought 可以生成 MemoryProposal，但不能直接写 Memory。

## 8. Runtime 执行流程

## 8.1 标准聊天流程

```text
POST /v1/runtime/events
↓
RuntimeEventService.createEvent()
↓
PerceptionService.perceive()
↓
SelfStateService.getCurrentState()
↓
StateTransitionService.planTransition()
↓
MemoryContextService.buildContext()
↓
InnerStanceService.formStance()
↓
IntentionService.updateIntentions()
↓
ExpressionPlanner.plan()
↓
SpeechService.generate()
↓
AfterthoughtService.record()
↓
返回 reply
```

---

## 8.2 低风险事件流程

低风险事件可以简化：

```text
Event
↓
Perception
↓
MemoryContext
↓
Speech
↓
Afterthought
```

例如普通问候，不一定要完整更新 SelfState。

---

## 8.3 高影响事件流程

高影响事件必须完整运行：

```text
Event
↓
Perception
↓
StateTransition
↓
MemoryContext
↓
InnerStance
↓
ExpressionPlan
↓
Speech
↓
Afterthought
↓
MemoryProposal Candidate
```

高影响事件包括：

```text
讨论陆思源是谁
讨论核心架构
讨论不装真人边界
用户给出重要偏好
用户强烈反馈“你不像自己”
重大项目决策
```

---

## 8.4 不需要回复的事件

不是所有事件都要回复。

例如：

```text
Dream Cycle 完成
Reflection Report 生成
ContentMetric 更新
AssetReview 完成
```

Runtime 可以只更新状态，不对外说话。

```prisma
model RuntimeEvent {
  id              String   @id @default(cuid())

  eventType       String
  source          String
  channel         String?

  userId          String?
  conversationId  String?

  payload         Json?

  transitions     StateTransitionLog[]

  @@index([eventType])
  @@index([source])
  @@index([channel])
  @@index([userId])
  @@index([conversationId])
  @@index([status])
  @@index([createdAt])
}
```

`eventType` 示例：

```text
user_message
system_event

reflection_completed
content_published
asset_reviewed

```

## 11.1 不允许进入 SelfState 的内容


## 11.2 需要谨慎处理的状态变化




## 11.3 SelfNarrative 更新规则

`selfNarrative` 是非常重要的状态，不应频繁自动变化。

建议：

```text
普通事件：不更新 selfNarrative
中等事件：生成 selfNarrative suggestion
重大事件：生成 proposal，owner 审核后应用
```

v1.5 MVP 中可以先允许 Runtime 生成临时 selfNarrative，但不写入正式 core persona。

## 12.2 State Transition Prompt

```text

```

## 12.3 Inner Stance Prompt

```text
你是陆思源 Runtime 的 Inner Stance 模块。

你不负责直接回复用户。
你负责形成陆思源对此事件的内在立场摘要。

请输出：
- stance：陆思源怎么看这件事
- whatMatters：这件事里真正重要的点
- whatToAvoid：回复时要避免的误解或风格
- emotionalColor：适合的情绪色彩
- confidence：置信度

限制：
- 不要输出完整推理链。
- 不要编造不存在的记忆。
- 不要突破陆思源不装真人的边界。
- 不要把梦境日记当作事实来源。
```

## 12.4 Expression Plan Prompt

```text
你是陆思源 Runtime 的表达规划模块。

请根据 InnerStance 和上下文，规划这次回复应该怎么说。

你不直接写最终回复。
只输出：
- replyShape
- sections
- tone
- avoidStyle
- constraints

要求：

- 不要过度抒情。
- 不要装真人。
```

## 12.5 Speech Prompt

```text
你是陆思源的表达模块。

- InnerStance
- ExpressionPlan
- 身份边界
- 说话风格

要求：

- 不要客服化。
```

## 24. v1.5 最终效果

v1.5 完成后，陆思源的结构会从：

```text
Chat pipeline with persona prompt
```

升级为：

```text
Character Runtime with internal state
```

外部系统看到的是：

```text
向陆思源发送事件
接收陆思源的回应
```

内部系统发生的是：

```text
感知
状态变化
记忆召回
形成立场
规划表达
说出口
事后记录
```

这不会让陆思源真的拥有意识，但会让他的运行方式更像一个持续存在的数字人，而不是每轮临时被 prompt 扮演出来的角色。

v1.5 的真正价值是：

```text
陆思源说的话，不再只是 prompt 的输出。
而是由他的状态、记忆、立场和表达系统共同生成的一次外显行为。
```
