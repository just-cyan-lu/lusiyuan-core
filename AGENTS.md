# Agent Working Rules

这份文件记录代码代理在本项目里长期要遵守的工作规则。

## 新功能边界

新增功能前，先检查它是否和已有功能重叠。

- 如果能复用已有模块，就复用已有模块。
- 如果职责交叉，先切清边界，必要时清理旧入口。
- 如果只是换个名字把已有功能做一遍，通常不要做。
- admin 页面里的新能力要能被人看懂，不只写代码入口。

当前例子：

- `RuntimeEventDetail` 解释“发生了什么、有没有资格影响长期状态”。
- `StateChangeDetail` 解释“最终写入后，状态实际改了什么”。
- 两者共用 admin 详情展示组件，避免重复造 UI。

## 开发期演进

项目还在开发阶段，没有正式上线，不需要为了旧数据库或旧代码路径做长期兼容。

- 遇到不兼容改动时，优先把设计改干净，删除或替换旧字段、旧入口、旧逻辑。
- 不要为了旧测试数据保留双轨逻辑；必要时之后清库重建。
- 如果新结构已经覆盖旧结构，直接迁移到新结构，不额外保留“兼容旧数据”的推断或 fallback。
- 只有当某个旧入口仍然承担真实功能时，才保留它；如果只是历史包袱，应当清理。
